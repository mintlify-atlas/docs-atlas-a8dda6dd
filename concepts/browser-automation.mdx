## The Core Problem

An AI agent interacting with a browser needs **sub-second latency** and **persistent state**.

If every command cold-starts a browser, you're waiting 3-5 seconds per tool call. For a QA session with 20+ commands, that's 60+ seconds of browser startup overhead.

Worse: if the browser dies between commands, you lose all state. Cookies, localStorage, login sessions, open tabs — all gone.

## The Solution: Persistent Daemon Architecture

gstack runs a **long-lived Chromium daemon** that the CLI talks to over localhost HTTP.

<Info>
First call starts everything (~3 seconds). Every call after that: ~100-200ms.
</Info>

### Architecture Diagram

```
Claude Code                     gstack
─────────                      ──────
                               ┌──────────────────────┐
  Tool call: $B snapshot -i    │  CLI (compiled binary)│
  ─────────────────────────→   │  • reads state file   │
                               │  • POST /command      │
                               │    to localhost:PORT   │
                               └──────────┬───────────┘
                                          │ HTTP
                               ┌──────────▼───────────┐
                               │  Server (Bun.serve)   │
                               │  • dispatches command  │
                               │  • talks to Chromium   │
                               │  • returns plain text  │
                               └──────────┬───────────┘
                                          │ CDP
                               ┌──────────▼───────────┐
                               │  Chromium (headless)   │
                               │  • persistent tabs     │
                               │  • cookies carry over  │
                               │  • 30min idle timeout  │
                               └───────────────────────┘
```

### Why This Design

**Persistent state:**
- Log in once, stay logged in
- Open a tab, it stays open
- localStorage persists across commands

**Sub-second commands:**
- After first call, every command is just an HTTP POST
- ~100-200ms round-trip including Chromium's work

**Automatic lifecycle:**
- Server auto-starts on first use
- Auto-shuts down after 30 minutes idle
- No process management needed

## The Ref System (@e1, @e2, @c1)

Refs are how the agent addresses page elements **without writing CSS selectors or XPath**.

### How It Works

```
1. Agent runs: $B snapshot -i
2. Server calls Playwright's page.accessibility.snapshot()
3. Parser walks ARIA tree, assigns sequential refs: @e1, @e2, @e3...
4. For each ref, builds Playwright Locator: 
   getByRole(role, { name }).nth(index)
5. Stores Map<string, Locator> on BrowserManager instance
6. Returns annotated tree as plain text

Later:
7. Agent runs: $B click @e3
8. Server resolves @e3 → Locator → locator.click()
```

### Why Locators, Not DOM Mutation

The obvious approach is to inject `data-ref="@e1"` attributes into the DOM. This breaks on:

- **CSP (Content Security Policy):** Many production sites block DOM modification from scripts
- **React/Vue/Svelte hydration:** Framework reconciliation can strip injected attributes
- **Shadow DOM:** Can't reach inside shadow roots from outside

Playwright Locators are **external to the DOM**. They use the accessibility tree (which Chromium maintains internally) and `getByRole()` queries. No DOM mutation, no CSP issues, no framework conflicts.

### Ref Types

| Flag | Ref Type | What It Finds | Example |
|------|----------|---------------|--------|
| `-i` | `@e1, @e2, ...` | Interactive elements (buttons, links, inputs) in ARIA tree | `$B click @e3` |
| `-C` | `@c1, @c2, ...` | Cursor-interactive elements (`cursor: pointer`, `onclick`, `tabindex`) not in ARIA tree | `$B click @c1` |

<Tip>
**Cursor-interactive refs** catch custom components that frameworks render as `<div>` but are actually buttons. Use `-C` when `-i` doesn't find what you need.
</Tip>

### Ref Lifecycle

Refs are **cleared on navigation** (the `framenavigated` event on the main frame).

This is by design — after navigation, all locators are stale. The agent must run `snapshot` again to get fresh refs. Stale refs should fail loudly, not click the wrong element.

### Snapshot Flags

All snapshot flags can be combined freely:

```bash
# Interactive elements only with refs
$B snapshot -i

# Compact (no empty structural nodes)
$B snapshot -c

# Limit depth
$B snapshot -d 3

# Scope to selector
$B snapshot -s ".modal"

# Diff against previous snapshot
$B snapshot -D

# Annotated screenshot with red overlay boxes
$B snapshot -i -a -o /tmp/annotated.png

# Find cursor-interactive elements too
$B snapshot -i -C
```

## Command Categories

Commands are categorized by side effects (defined in `browse/src/commands.ts`):

### READ Commands
No mutations. Safe to retry. Returns page state.

```bash
$B text                    # Page text
$B html                    # Full page HTML
$B links                   # All links
$B console                 # Console messages
$B network                 # Network requests
$B cookies                 # All cookies
$B forms                   # Form fields as JSON
$B accessibility           # Full ARIA tree
```

### WRITE Commands
Mutates page state. Not idempotent.

```bash
$B goto <url>              # Navigate
$B click <sel>             # Click element
$B fill <sel> <val>        # Fill input
$B press <key>             # Press key
$B type <text>             # Type into focused element
$B upload <sel> <file>     # Upload file
$B scroll [sel]            # Scroll element or page
$B select <sel> <val>      # Select dropdown option
$B hover <sel>             # Hover element
```

### META Commands
Server-level operations.

```bash
$B snapshot [flags]        # Accessibility tree + refs
$B screenshot [path]       # Take screenshot
$B tabs                    # List tabs
$B tab <id>                # Switch tab
$B newtab [url]            # Open new tab
$B closetab [id]           # Close tab
$B chain                   # Run commands from JSON stdin
$B status                  # Health check
$B stop                    # Shutdown server
$B restart                 # Restart server
```

<Note>
The server uses command categories for dispatch routing. See `ARCHITECTURE.md` for implementation details.
</Note>

## State Persistence

### State File

The server writes `.gstack/browse.json` (atomic write via tmp + rename, mode 0o600):

```json
{
  "pid": 12345,
  "port": 34567,
  "token": "uuid-v4",
  "startedAt": "2026-03-14T...",
  "binaryVersion": "abc123"
}
```

The CLI reads this file to find the server. If the file is missing, stale, or the PID is dead, the CLI spawns a new server.

### Port Selection

Random port between 10000-60000 (retry up to 5 on collision).

<Info>
This means **10 Conductor workspaces** can each run their own browse daemon with zero configuration and zero port conflicts.
</Info>

### Version Auto-Restart

The build writes `git rev-parse HEAD` to `browse/dist/.version`.

On each CLI invocation, if the binary's version doesn't match the running server's `binaryVersion`, the CLI kills the old server and starts a new one.

This prevents "stale binary" bugs entirely — rebuild the binary, next command picks it up automatically.

## Cookie Handling

### Security Model

Cookies are the most sensitive data gstack handles.

**1. Keychain access requires user approval**
- First cookie import per browser triggers macOS Keychain dialog
- User must click "Allow" or "Always Allow"
- gstack never silently accesses credentials

**2. Decryption happens in-process**
- Cookie values decrypted in memory (PBKDF2 + AES-128-CBC)
- Loaded into Playwright context
- Never written to disk in plaintext

**3. Database is read-only**
- Copies Chromium cookie DB to temp file (avoids SQLite lock conflicts)
- Opens read-only
- Never modifies real browser's cookie database

**4. Key caching is per-session**
- Keychain password + derived AES key cached in memory
- Cache cleared when server shuts down

**5. No cookie values in logs**
- Console, network, dialog logs never contain cookie values
- `cookies` command outputs metadata (domain, name, expiry) but values are truncated

### Import Flow

```bash
# Interactive picker UI
$B cookie-import-browser

# Direct domain import
$B cookie-import-browser comet --domain .github.com

# Supported browsers
# Comet, Chrome, Arc, Brave, Edge (auto-detected)
```

The cookie picker UI never displays cookie values — only domain names and counts.

## Logging Architecture

Three ring buffers (50,000 entries each, O(1) push):

```
Browser events → CircularBuffer (in-memory) → Async flush to .gstack/*.log
```

- **Console messages** → `.gstack/console.log`
- **Network requests** → `.gstack/network.log`
- **Dialog events** → `.gstack/dialog.log`

### Why Ring Buffers

- HTTP request handling never blocked by disk I/O
- Logs survive server crashes (up to 1 second data loss)
- Memory is bounded (50K entries × 3 buffers)
- Disk files are append-only, readable by external tools

The `console`, `network`, and `dialog` commands read from **in-memory buffers**, not disk. Disk files are for post-mortem debugging.

## Error Philosophy

Errors are for AI agents, not humans. Every error message must be **actionable**:

<CodeGroup>

```text Bad Error
Element not found
```

```text Good Error
Element not found or not interactable. 
Run `snapshot -i` to see available elements.
```

</CodeGroup>

<CodeGroup>

```text Bad Error
Selector matched multiple elements
```

```text Good Error  
Selector matched multiple elements. 
Use @refs from `snapshot` instead.
```

</CodeGroup>

<CodeGroup>

```text Bad Error
Timeout
```

```text Good Error
Navigation timed out after 30s. 
The page may be slow or the URL may be wrong.
```

</CodeGroup>

Playwright's native errors are rewritten through `wrapError()` to strip internal stack traces and add guidance. The agent should be able to read the error and know what to do next without human intervention.

### Crash Recovery

The server **doesn't try to self-heal**. If Chromium crashes (`browser.on('disconnected')`), the server exits immediately.

The CLI detects the dead server on the next command and auto-restarts. This is simpler and more reliable than trying to reconnect to a half-dead browser process.

## Why Bun

Node.js would work. Bun is better here for four reasons:

### 1. Compiled Binaries

`bun build --compile` produces a single ~58MB executable. No `node_modules` at runtime, no `npx`, no PATH configuration. The binary just runs.

This matters because gstack installs into `~/.claude/skills/` where users don't expect to manage a Node.js project.

### 2. Native SQLite

Cookie decryption reads Chromium's SQLite cookie database directly. Bun has `new Database()` built in — no `better-sqlite3`, no native addon compilation, no gyp.

One less thing that breaks on different machines.

### 3. Native TypeScript

The server runs as `bun run server.ts` during development. No compilation step, no `ts-node`, no source maps to debug.

The compiled binary is for deployment; source files are for development.

### 4. Built-in HTTP Server

`Bun.serve()` is fast, simple, and doesn't need Express or Fastify. The server handles ~10 routes total. A framework would be overhead.

<Note>
The bottleneck is always Chromium, not the CLI or server. Bun's startup speed (~1ms for compiled binary vs ~100ms for Node) is nice but not the reason we chose it. The compiled binary and native SQLite are.
</Note>

## What's Intentionally Not Here

**No WebSocket streaming**
- HTTP request/response is simpler, debuggable with curl, fast enough
- Streaming would add complexity for marginal benefit

**No MCP protocol**
- MCP adds JSON schema overhead per request
- Requires persistent connection
- Plain HTTP + plain text output is lighter on tokens and easier to debug

**No multi-user support**
- One server per workspace, one user
- Token auth is defense-in-depth, not multi-tenancy

**No Windows/Linux cookie decryption**
- macOS Keychain is the only supported credential store
- Linux (GNOME Keyring/kwallet) and Windows (DPAPI) are architecturally possible but not implemented

**No iframe support**
- Playwright can handle iframes
- The ref system doesn't cross frame boundaries yet
- Most-requested missing feature

## Security Model

### Localhost Only

HTTP server binds to `localhost`, not `0.0.0.0`. Not reachable from network.

### Bearer Token Auth

Every server session generates a random UUID token, written to state file with mode 0o600 (owner-only read).

Every HTTP request must include `Authorization: Bearer <token>`. If token doesn't match, server returns 401.

This prevents other processes on the same machine from talking to your browse server.

Exemptions: `/cookie-picker` and `/health` (localhost-only, don't execute commands).

### Shell Injection Prevention

Browser registry (Comet, Chrome, Arc, Brave, Edge) is hardcoded. Database paths constructed from known constants, never from user input.

Keychain access uses `Bun.spawn()` with explicit argument arrays, not shell string interpolation.

## Multi-Workspace Isolation

With [Conductor](https://conductor.build) running 10 parallel Claude Code sessions:

- Each workspace gets its own isolated browser instance
- Separate Chromium process per workspace
- Cookies, tabs, logs stored in `.gstack/` inside each project root
- No port collisions (random port 10000-60000)
- No shared state
- No configuration needed

`/browse` and `/qa` sessions never interfere with each other, even across ten parallel workspaces.
