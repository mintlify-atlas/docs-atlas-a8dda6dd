## Overview

The `/qa` skill turns QA from "go click around manually" into systematic, automated testing with evidence. It gives Claude eyes via the headless browser and a testing methodology that automatically scopes to what changed.

<Note>
**Most common use case:** You're on a feature branch, you just finished coding, and you want to verify everything works. Just say `/qa` — it reads your git diff, identifies which pages your changes affect, and tests each one. No URL required. No manual test plan.
</Note>

## Four QA Modes

<Tabs>
  <Tab title="Diff-Aware (Auto)">
    **Automatic when on a feature branch with no URL.**

    This is the **primary mode** for developers verifying their work.

    ### How It Works

    <Steps>
      <Step title="Analyze branch diff">
        ```bash
        git diff main...HEAD --name-only
        git log main..HEAD --oneline
        ```

        Understands what changed — controllers, views, models, styles, API endpoints.
      </Step>

      <Step title="Identify affected pages">
        - Controller/route files → which URL paths they serve
        - View/component files → which pages render them  
        - Model/service files → which pages use those models
        - CSS files → which pages include those stylesheets
        - API endpoints → test them with `fetch()`
      </Step>

      <Step title="Detect running app">
        ```bash
        # Checks common dev ports automatically
        localhost:3000, :4000, :8080, etc.
        ```

        If no local app found, checks for staging/preview URL in PR.
      </Step>

      <Step title="Test affected routes">
        For each page:
        - Navigate and take screenshot
        - Check console for errors
        - If interactive (forms, buttons), test end-to-end
        - Use snapshot diff to verify expected changes
      </Step>

      <Step title="Cross-reference intent">
        Reads commit messages and PR description to verify the change does what it's supposed to do.
      </Step>
    </Steps>

    ### Example Output

    ```bash
    You:   /qa

    Claude: Analyzing branch diff against main...
            12 files changed: 3 controllers, 2 views, 4 services, 3 tests

            Affected routes: /listings/new, /listings/:id, /api/listings
            Detected app running on localhost:3000.

            [Tests each affected page — navigates, fills forms, clicks buttons,
            screenshots, checks console errors]

            QA Report: 3 routes tested, all working.
            - /listings/new: upload + enrichment flow works end to end
            - /listings/:id: detail page renders correctly  
            - /api/listings: returns 200 with expected shape
            No console errors. No regressions on adjacent pages.
    ```

    <Note>
    Changes tested: **N pages/routes affected by this branch**. Scoped to what you actually changed, not the entire app.
    </Note>
  </Tab>

  <Tab title="Full">
    **Default when URL is provided.**

    Systematic exploration of the entire app. Visit every reachable page. Document 5-10 well-evidenced issues. Produce health score.

    **Takes:** 5-15 minutes depending on app size

    ### Example

    ```bash
    You:   /qa https://staging.myapp.com

    Claude: [Explores 12 pages, fills 3 forms, tests 2 flows]

            QA Report: staging.myapp.com — Health Score: 72/100

            Top 3 Issues:
            1. CRITICAL: Checkout form submits with empty required fields
            2. HIGH: Mobile nav menu doesn't close after selecting an item  
            3. MEDIUM: Dashboard chart overlaps sidebar below 1024px

            [Full report with screenshots saved to .gstack/qa-reports/]
    ```

    ### What It Tests

    At each page, follows the **per-page exploration checklist**:

    1. **Visual scan** — Layout issues in annotated screenshot
    2. **Interactive elements** — Click buttons, links, controls  
    3. **Forms** — Fill and submit with empty, invalid, edge cases
    4. **Navigation** — Check all paths in and out
    5. **States** — Empty state, loading, error, overflow
    6. **Console** — Any JS errors after interactions?
    7. **Responsiveness** — Mobile viewport (375x812)

    <Note>
    Depth judgment: Spends more time on core features (homepage, checkout, search) and less on secondary pages (about, terms).
    </Note>
  </Tab>

  <Tab title="Quick">
    **Use `--quick` flag for 30-second smoke test.**

    Visit homepage + top 5 navigation targets. Check: page loads? Console errors? Broken links?

    ### Example

    ```bash
    You:   /qa https://staging.myapp.com --quick

    Claude: [Smoke test: homepage + 5 pages, 30 seconds]
            Health Score: 91/100. No critical issues. 
            1 medium: mobile nav overlap.
    ```

    **When to use:**
    - After deployments (quick sanity check)
    - Before important demos
    - Monitoring staging health

    **What it skips:** Per-page checklist, form testing, detailed issue documentation.
  </Tab>

  <Tab title="Regression">
    **Use `--regression <baseline>` to compare against previous run.**

    Run full mode, then load baseline from previous QA session. Diff: which issues are fixed? Which are new? What's the score delta?

    ### Example

    ```bash
    You:   /qa https://staging.myapp.com --regression .gstack/qa-reports/baseline.json

    Claude: Health Score: 78/100 (was 72, +6)

            Fixed since baseline:
            - CRITICAL: Checkout form validation (was ISSUE-001)
            - HIGH: Mobile nav close behavior (was ISSUE-003)

            New issues:
            - MEDIUM: Search filters not clearing on page change

            Unchanged:
            - MEDIUM: Dashboard chart overlap (ISSUE-004)
    ```

    **Baseline format:** Saved automatically at `.gstack/qa-reports/baseline.json` after every full run.
  </Tab>
</Tabs>

## Health Score Rubric

Computes each category score (0-100), then takes weighted average for final score.

### Category Scoring

<CodeGroup>
```yaml Console (15% weight)
0 errors → 100
1-3 errors → 70  
4-10 errors → 40
10+ errors → 10
```

```yaml Links (10% weight)
0 broken → 100
Each broken link → -15 (minimum 0)
```

```yaml Per-Category Deductions
Visual (10%), Functional (20%), UX (15%),
Performance (10%), Content (5%), Accessibility (15%)

Each starts at 100, then:
Critical issue → -25
High issue → -15
Medium issue → -8  
Low issue → -3
Minimum 0 per category
```
</CodeGroup>

### Final Score Formula

```
score = Σ (category_score × weight)
```

**Example calculation:**

| Category | Score | Weight | Contribution |
|----------|-------|--------|-------------|
| Console | 70 | 15% | 10.5 |
| Links | 100 | 10% | 10.0 |
| Visual | 85 | 10% | 8.5 |
| Functional | 50 | 20% | 10.0 |
| UX | 75 | 15% | 11.25 |
| Performance | 90 | 10% | 9.0 |
| Content | 95 | 5% | 4.75 |
| Accessibility | 60 | 15% | 9.0 |
| **Total** | | **100%** | **73** |

## Report Structure

Every QA run produces a structured markdown report:

```
.gstack/qa-reports/
├── qa-report-{domain}-{YYYY-MM-DD}.md
├── screenshots/
│   ├── initial.png
│   ├── issue-001-step-1.png
│   ├── issue-001-result.png
│   └── ...
└── baseline.json
```

### Report Template

Each report includes:

<Steps>
  <Step title="Metadata">
    - Date, duration, pages visited, screenshot count
    - Framework detected (Next.js, Rails, WordPress, SPA)
    - Health score
  </Step>

  <Step title="Top 3 Things to Fix">
    The 3 highest-severity issues with clear impact
  </Step>

  <Step title="Issue Documentation">
    Each issue has:
    - Severity (Critical, High, Medium, Low)
    - Category (Visual, Functional, UX, Performance, Content, Accessibility, Console)
    - Reproduction steps
    - Screenshot evidence (before/after for interactive bugs)
    - File references when known
  </Step>

  <Step title="Console Health Summary">
    Aggregate of all console errors across pages
  </Step>

  <Step title="Severity Counts Table">
    Breakdown by category and severity
  </Step>
</Steps>

## Framework-Specific Guidance

The `/qa` skill detects your framework and adjusts its testing strategy:

<Tabs>
  <Tab title="Next.js">
    **Detected by:** `__next` in HTML or `_next/data` requests

    **What it checks:**
    - Console for hydration errors (`Hydration failed`, `Text content did not match`)
    - `_next/data` requests — 404s indicate broken data fetching
    - Client-side navigation (clicks links, doesn't just `goto`) — catches routing issues
    - CLS (Cumulative Layout Shift) on pages with dynamic content
  </Tab>

  <Tab title="Rails">
    **Detected by:** `csrf-token` meta tag

    **What it checks:**
    - N+1 query warnings in console (if development mode)
    - CSRF token presence in forms
    - Turbo/Stimulus integration — page transitions smooth?
    - Flash messages appearing and dismissing correctly
  </Tab>

  <Tab title="WordPress">
    **Detected by:** `wp-content` in URLs

    **What it checks:**
    - Plugin conflicts (JS errors from different plugins)
    - Admin bar visibility for logged-in users
    - REST API endpoints (`/wp-json/`)
    - Mixed content warnings (common with WP)
  </Tab>

  <Tab title="SPA (React/Vue/Angular)">
    **Detected by:** Client-side routing, no page reloads

    **What it checks:**
    - Uses `snapshot -i` for navigation (not `links` command)
    - Stale state — navigate away and back, does data refresh?
    - Browser back/forward — does app handle history correctly?
    - Memory leaks (monitor console after extended use)
  </Tab>
</Tabs>

## Real Command Usage

From `qa/SKILL.md` — how the skill actually works:

### Setup Phase

```bash
# Find the browse binary
_ROOT=$(git rev-parse --show-toplevel 2>/dev/null)
B=""
[ -n "$_ROOT" ] && [ -x "$_ROOT/.claude/skills/gstack/browse/dist/browse" ] && B="$_ROOT/.claude/skills/gstack/browse/dist/browse"
[ -z "$B" ] && B=~/.claude/skills/gstack/browse/dist/browse

# Create output directories  
REPORT_DIR=".gstack/qa-reports"
mkdir -p "$REPORT_DIR/screenshots"
```

### Diff-Aware Analysis

```bash
# Analyze branch diff
git diff main...HEAD --name-only
git log main..HEAD --oneline

# Detect running app
$B goto http://localhost:3000 2>/dev/null && echo "Found app on :3000" || \
$B goto http://localhost:4000 2>/dev/null && echo "Found app on :4000" || \
$B goto http://localhost:8080 2>/dev/null && echo "Found app on :8080"
```

### Testing Each Page

```bash
# Navigate and capture
$B goto <page-url>
$B snapshot -i -a -o "$REPORT_DIR/screenshots/page-name.png"
$B console --errors

# Test interactive elements
$B screenshot "$REPORT_DIR/screenshots/issue-001-step-1.png"
$B click @e5
$B screenshot "$REPORT_DIR/screenshots/issue-001-result.png"  
$B snapshot -D  # Show what changed
```

### Responsive Testing

```bash
$B viewport 375x812
$B screenshot "$REPORT_DIR/screenshots/page-mobile.png"
$B viewport 1280x720
```

## Important Rules

<Warning>
**Repro is everything.** Every issue needs at least one screenshot. No exceptions.
</Warning>

1. **Verify before documenting** — Retry the issue once to confirm it's reproducible, not a fluke.

2. **Never include credentials** — Write `[REDACTED]` for passwords in repro steps.

3. **Write incrementally** — Append each issue to the report as you find it. Don't batch.

4. **Never read source code** — Test as a user, not a developer.

5. **Check console after every interaction** — JS errors that don't surface visually are still bugs.

6. **Test like a user** — Use realistic data. Walk through complete workflows end-to-end.

7. **Depth over breadth** — 5-10 well-documented issues with evidence > 20 vague descriptions.

8. **Use `snapshot -C` for tricky UIs** — Finds clickable divs that the accessibility tree misses.

<CodeGroup>

```bash Chrome/Arc/Brave
/setup-browser-cookies staging.myapp.com

# Auto-detects installed browsers
# Decrypts cookies via macOS Keychain  
# Imports into headless session
```

```bash From JSON File
$B cookie-import cookies.json
```
</CodeGroup>

After import, `/qa` can test pages behind login without manual authentication.
